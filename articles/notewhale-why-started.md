# Building NoteWhale: Why I Started

> From a Student's Frustration to an AI-Powered Learning Platform

## 起点

南京大学软件工程的课程密度很高。一个学期下来，我的笔记散落在至少五个地方：课堂用的 Notion、代码实验的 GitHub README、考前临时整理的 Word 文档、和同学协作的飞书文档、还有手写的草稿纸。

每次我想找一个知识点——比如"操作系统页面置换算法"——我都得在五六个工具之间来回搜索，还不一定能找到。更 frustrating 的是，这些笔记之间没有任何关联。操作系统里的"虚拟内存"和计算机网络的"分页"明明有概念上的联系，但没有任何工具帮我发现这种联系。

这就是 NoteWhale 的起点：一个简单的想法——如果笔记能自动分类、自动关联、还能语义搜索呢？

## 产品定义

NoteWhale 是一个 AI 驱动的知识管理平台。核心功能很直接：

- **自动分类与标签**：写入笔记后，LLM 自动提取关键词、分类、打标签
- **语义搜索**：不是关键词匹配，而是理解意图的向量检索
- **知识图谱**：笔记之间自动发现关联，形成可视化的知识网络

技术栈是 React + Node.js + Python + LLM + RAG。前端用 React 做交互，后端 Node.js 处理 API，Python 跑模型和向量化。语义搜索的核心是 [[rag-pipeline]]——把笔记分块、向量化、存入向量数据库，查询时用相似度检索找到最相关的内容。

## 为什么不直接用现成工具

Notion 很好，但它不帮你关联笔记。Obsidian 有关联，但语义搜索弱。Roam Research 有双向链接，但太贵了。

我不是说这些工具不好。我是说，作为一个软件工程的学生，我想要一个能完全控制、能深度定制、能和我的学习流程紧密集成的工具。而且，我想亲手实践 [[rag-pipeline]] 和 [[llm-basics]] 里学到的概念——不只是看论文，而是真正用起来。

这其实是一个 [[product-thinking]] 的练习：从用户痛点出发（我自己的痛点），而不是从技术能力出发。

## MVP 设计

按照 [[mvp-design]] 的原则，第一版只做三件事：

1. **写入笔记**：支持 Markdown，支持快速录入
2. **自动标签**：LLM 读取内容，自动提取关键词和分类
3. **语义搜索**：输入一个问题，返回最相关的笔记片段

知识图谱、协作、离线模式——这些都是后续迭代的事。MVP 的核心价值是：让笔记可搜索、可关联，而且这个过程是自动的。

## 技术挑战

最大的挑战不是前端或后端，而是 RAG 管道的设计。

笔记的分块策略很关键。按段落分？按语义分？分块太大，检索精度低；分块太小，上下文丢失。这些是我目前在研究的问题——还没有确定的方案，需要在实际数据上实验后才能下结论。

向量数据库的选型也是开放问题。需求是：本地能跑、支持相似度搜索、不需要太多运维。目前还在调研阶段。

## 和学习的关系

NoteWhale 不只是一个项目，它是我学习的一种方式。

通过构建它，我深入理解了 RAG 管道的每一步：文档分块、向量化、相似度检索、上下文注入、生成回答。这些概念在 wiki 里只是几行文字，但真正实现一遍，理解完全不同。

这和我在 [[agent-overview]] 里记录的 Agent 实践是同一条路径：从概念理解到工程实现。

## 总结

NoteWhale 还在早期阶段。MVP 还没跑通，分块策略和向量数据库的选型都还在调研。但方向是清晰的：从学生痛点出发，用 AI 解决知识管理的问题，同时在这个过程中把 RAG、LLM、产品设计的理论变成实践。

不追求一步到位，先做最小可用版本，再用真实使用反馈驱动迭代。
